🕒 TG Status Updater
File: cron/src/handlers/tg-status-updater.js Schedule: cron(0 4 * * ? *) — 04:00 UTC daily Lambda: {NamePrefix}-tg-status-updater-{Environment}
🎯 Purpose
Automatically advances Test Guideline (TG) status codes based on deadline dates stored in the TG table. Handles six transitions across both the TWP pipeline (existing) and the newly added TC-EDC pipeline. No emails are sent — this is a pure database mutation job.
⚙️ Functionality — Status Transitions
Repository Function SQL Condition Transition Pipeline
updateToLEDraft() LE_Draft_StartDate = CURRENT_DATE
AND Status_Code = 'CRT'
AND IsTGStatusOverride = 'N'		 CRT → LED TWP
updateToIEComments() IE_Comments_StartDate = CURRENT_DATE
AND Status_Code = 'LED'
AND IsTGStatusOverride = 'N'		 LED → IEC TWP
updateToLEChecking() LE_Checking_StartDate = CURRENT_DATE
AND Status_Code = 'IEC'
AND IsTGStatusOverride = 'N'		 IEC → LEC TWP
updateToSentToUpov() DATE(LE_Checking_EndDate) + INTERVAL 1 DAY = CURRENT_DATE
AND Status_Code = 'LEC'
AND IsTGStatusOverride = 'N'		 LEC → STU + sets CPI_date TWP
updateToEccComments() 🆕 ECC_StartDate = CURRENT_DATE
AND Status_Code = 'TCD'
AND CPI_TechWorkParty IN ('TC','TC-EDC')
AND IsTGStatusOverride = 'N'		 TCD → ECC TC-EDC
updateToStuFromEcc() 🆕 DATE(ECC_EndDate) + INTERVAL 1 DAY = CURRENT_DATE
AND Status_Code = 'ECC'
AND IsTGStatusOverride = 'N'		 ECC → STU TC-EDC

🆕 Classification
Partially New

TWP transitions (CRT→LED, LED→IEC, IEC→LEC, LEC→STU) are existing. TC-EDC transitions via updateToEccComments() and updateToStuFromEcc() are newly implemented.

⚠️ Issues
ISSUE 1 HIGH
No retry / partial-failure recovery. If any UPDATE throws (e.g. DB connectivity loss), the handler re-throws, aborting all remaining transitions for that day. TGs that should have been promoted will stay in their old status until the next 04:00 UTC run — a 24-hour lag with no automated recovery.
ISSUE 2 MEDIUM
CPI_date not set on TC-EDC path. updateToSentToUpov() sets CPI_date = CURRENT_TIMESTAMP when LEC→STU. The new updateToStuFromEcc() (ECC→STU) does not set CPI_date. If CPI_date is used downstream as a "completed pipeline" marker, TC-EDC TGs will have a null value.
ISSUE 3 LOW
findTGsWithDateToday() is dead-weight diagnostic code. The result is only logged; it does not filter the subsequent UPDATEs. If removed, behaviour is identical. Also, this diagnostic SELECT is missing ECC_StartDate awareness — it checks LE_Draft_EndDate and LE_Checking_EndDate but not ECC_EndDate in one of its OR branches.
ISSUE 4 LOW
Silent skip on IsTGStatusOverride = 'Y'. Overridden TGs are silently excluded from all transitions. No log entry is emitted per skipped TG. Debugging override-related issues in production requires manual DB inspection.

🕒 IE Status Updater
File: cron/src/handlers/ie-status-updater.js Schedule: cron(5 4 * * ? *) — 04:05 UTC daily Lambda: {NamePrefix}-ie-status-updater-{Environment}
🎯 Purpose
After Job #1 promotes TGs to LEC or LES, this job deactivates all Interested Expert (IE) role assignments on those TGs by setting Tg_Users.Status_Code = 'I' (Inactive). The 5-minute offset from Job #1 ensures newly transitioned TGs are already in LEC/LES before this SELECT runs.
⚙️ Functionality
StepFunctionSQLFile
1 findTGIdsByStatus(['LEC','LES']) SELECT TG_ID FROM TG WHERE Status_Code IN ('LEC','LES') cron/src/repositories/tg-user.js
2 Early-exit if empty — cron/src/handlers/ie-status-updater.js
3 deactivateIEUsers(ids) UPDATE Tg_Users SET Status_Code = 'I' WHERE TG_ID IN (...) AND Role_Code = 'IE' cron/src/repositories/tg-user.js

🧾 Templates Used

🆕 Classification
Existing

No new code added. Not referenced as a new feature in requirements documents.

⚠️ Issues
ISSUE 1 HIGH
Timing dependency on Job #1 with no hard guarantee. The 5-minute gap is hardcoded in cron.yml. If Job #1 takes longer than 5 minutes (slow DB, large dataset), this job runs before Job #1 finishes, missing newly-transitioned TGs. No explicit Lambda chaining or Step Function coordination exists.
ISSUE 2 MEDIUM
Re-deactivates IEs on all LEC/LES TGs daily, not just new ones. Every day it issues an UPDATE on every TG in LEC/LES — including those transitioned days/weeks ago whose IEs were already deactivated. Idempotent but produces unnecessary writes and no-op DB round-trips.
ISSUE 3 LOW
Unbounded IN clause. deactivateIEUsers(ids) builds WHERE TG_ID IN (?, ?, ...) dynamically. In a production system with many TGs in LEC/LES, this clause can grow very large, causing potential DB performance degradation or query plan issues.

🆕 Classification
Partially New

TWP reminders (LE Draft, IE Comments, LE Checking) are existing. The sendEccReminders() function, the eccPhase result bucket, and the findEccStartReminders() / findEccEndReminders() repository functions are newly implemented for the TC-EDC pipeline.

⚠️ Issues
ISSUE 1 MEDIUM
findEccStartReminders uses wrong status check. The start reminder queries for Status_Code = 'TCD', meaning it fires while the TG is still in pre-ECC state. This is technically correct because Job #1 runs at 04:00 and transitions TCD → ECC before Job #3 runs at 04:10 — but it creates a fragile coupling. If Job #1 is delayed, the ECC start reminder fires while the TG is still TCD.
ISSUE 2 MEDIUM
ECC recipient query uses global latest-session subquery. Both findEccStartReminders and findEccEndReminders use (SELECT tb.TB_CodeID FROM technical_body tb WHERE tb.TB_Code IN ('TC','TC-EDC') ORDER BY tb.TB_Year DESC LIMIT 1) — this always picks the most recent TC/TC-EDC session globally. If a TG belongs to an older session, EDC members of the newest session receive a reminder for a TG they may not be reviewing.
ISSUE 3 MEDIUM
Hardcoded BCC and app URL in source code. bccEmail = 'tgtemplate@upov.int' and the link https://www3.wipo.int/upovtg are hardcoded with no env var override. Changing BCC or the URL for a different environment requires a code change and redeploy.
ISSUE 4 MEDIUM
No separate email template files — inline HTML only. Email body is a template literal inside the handler. No templating engine, no localisation support, no text/plain fallback (SES sends HTML-only). Changing copy requires a code deploy.
ISSUE 5 LOW
Deduplication merges start + end reminder TGs into one email without labelling. If a user has one TG starting today and another TG ending in 7 days, both appear in the same email list without any label distinguishing "starts today" vs "ends soon". Recipients can't tell which TG is urgent.
ISSUE 6 LOW
No in-app notifications. No notifications table writes, no WebSocket events, no push notifications. The system is email-only. Users who miss the email have no in-app indication of upcoming deadlines.
